# MIRA — Maintenance Intelligence & Reporting Assistant (local/private prototype)

MIRA is a **local, private assistant layer** inside the maintenance dashboard. It
answers maintenance questions and drafts reports using the **same KPI outputs the
dashboard already calculates** — it never re-implements KPI maths and never reads
raw Excel / D365 data directly.

> **Note:** Training a full Transformer model from scratch is **not** included in
> this prototype. MIRA is a controlled local assistant layer using dashboard KPI
> tools. A local LLM or approved AI service can be connected later if approved by IT.

## Data flow

```
imported MR/WO data
  → existing dashboard processing / KPI logic        (downtime_service, maintenance_service, pm_schedule_service)
  → existing dashboard KPI outputs
  → MIRA kpiQueryService        (services/kpi_query_service.py)   read-only
  → MIRA privacyGuardService    (privacy/privacy_guard_service.py) scrub / cap / label
  → MIRA mockMiraProvider       (providers/mock_provider.py)       rule-based, offline
  → MIRA response / report draft
```

## Modes

- **KPI Summary Mode (default):** aggregated KPI outputs only. No row-level data.
- **Limited Filtered Rows Mode:** used only for explicit work-order search. Backend
  filters first, returns **10–25 rows max**, field-reduced via an allowlist, with
  staff/vendor names and free-text remarks removed. Never the full dataset.

snake_case primary names; camelCase aliases provided to match the spec.

| Function | Reuses |

| `get_work_orders` / `getWorkOrders` | `build_downtime_payload` rows (limited) |

subAssetGroup, maintenanceType, status, mappingStatus` (see `core/context.py`).

## Privacy guard guarantees (`privacy/privacy_guard_service.py`)

- Default-deny allowlist for WO rows (drops `started_by`, `created_by`,
  `description`, remarks, vendor/cost…).
- Row cap 10–25 (`config.ROW_CAP_MAX`); `block_full_dataset()` refuses large dumps.
- `redact_secrets()` strips anything resembling an API key / token.
- `mark_draft()` stamps **“Draft generated by MIRA for review”** on every output.
- `console_safe: true` flag + no raw rows in summary payloads (frontend should not
  `console.log` full records).

## Providers

- **`mock_provider` (default):** deterministic, offline, rule-based templates.
- **`local_llm_stub` (disabled):** inert seam for a future **IT-approved local**
  model runner. Does not import or connect to Ollama / any model. Enable only with
  `MIRA_PROVIDER=local` **and** `MIRA_LOCAL_LLM_ENABLED=true` after IT approval.

## Routes (`mira/api.py`, prefix `/api/mira`)

| Route | Purpose |
|---|---|
| `GET /health` | status, provider, caps |
| `GET/POST /summary` | consolidated KPI snapshot |
| `POST /query` | free-text question → intent → KPI summary (or limited rows) |
| `GET/POST /mttr`, `/mtbf`, `/pm-schedule`, `/data-quality`, `/stage-compare` | single KPIs |
| `GET/POST /work-orders` | Limited Filtered Rows Mode (capped, scrubbed) |
| `GET/POST /report-draft` | monthly report draft (markdown + structured) |

Each route: read filters → KPI function → **privacy guard** → **provider** → JSON.
Registered in `app.py` via a single guarded `app.register_blueprint(mira_bp)`.

## How to run

```bash
cd backend

# 1) Smoke tests (no server needed)
python -m mira.tests.test_mira

# 2) Run the dashboard (MIRA is mounted automatically)
python app.py
#    then, in another shell:
curl "http://localhost:5005/api/mira/health"
curl "http://localhost:5005/api/mira/summary?stage=all&year=2026"
curl -X POST "http://localhost:5005/api/mira/query" \
     -H "Content-Type: application/json" \
     -d '{"question":"What is the MTTR this month?","filters":{"stage":"all","year":2026,"month":6}}'
```

Restart the Flask backend after Python API/schema changes. The browser can load
new JavaScript immediately, but `/api/mira/*` continues serving the old Python
process until Flask is restarted. Check `/api/mira/health` for the active
`backend_version`, startup timestamp, provider status, and available MIRA routes.

Provider/limits are env-overridable but default to the safe values (no `.env`
needed): `MIRA_PROVIDER=mock`, `MIRA_LOCAL_LLM_ENABLED=false`, `MIRA_ROW_CAP_MAX=25`.

## Assumptions

- `src/mira` → `backend/mira` (repo uses `backend/`, not `src/`).
- KPI access is **in-process** (calls the builder functions directly), which reuses
  their caches and guarantees MIRA numbers equal dashboard numbers.
- “Closed” work orders = `total − open` from the same downtime summary.
- A window with no imported WO data (e.g. a future month) correctly returns
  `0` / “not available” — MIRA mirrors the dashboard, it does not fabricate values.
- MIRA mirrors dashboard KPI definitions verbatim (e.g. `requires_attention_count`),
  so its figures never diverge from the dashboard.
